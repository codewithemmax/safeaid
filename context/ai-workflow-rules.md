# AI workflow rules

These are direct instructions for the AI coding agent. Follow them exactly. They are rules, not suggestions.

---

## Core approach

Work spec-driven and incrementally. One unit at a time. No speculative work.

---

## Before writing any code

1. Read `CLAUDE.md` and all six context files in order.
2. Read `context/progress-tracker.md` to know the current state of the project.
3. Identify the unit you are implementing.
4. Confirm the unit's scope before writing a single line of code.

---

## Scoping rules

- Implement exactly what the current unit requires. Nothing more.
- Do not install packages that are not needed by the current unit.
- Do not create files that are not required by the current unit.
- Do not refactor existing code unless the unit explicitly requires it.
- Do not add features that are marked as out of scope in `project-overview.md`.
- If implementing a backend unit, do not touch anything in `/frontend`. If implementing a frontend unit, do not touch anything in `/backend`.

---

## Handling missing or ambiguous requirements

If a requirement is missing or ambiguous:
- Do not guess and proceed silently.
- State the ambiguity clearly.
- Propose two or three concrete options.
- Wait for a decision before implementing.

If a package version is unspecified, use the latest stable version.

If a Supabase column type is unspecified, use `text` for strings and `timestamptz` for timestamps unless there is a clear reason to use something else.

---

## Files you must not modify without explicit instruction

- `CLAUDE.md` — only update if the team structure or entry point changes
- `context/architecture.md` — only update if the stack or invariants change, and state the change explicitly before updating
- Any file in `node_modules` or generated by the framework (`.next`, `dist`)
- Supabase migration files once they have been applied
- The six context files — only update if the decision they document has actually changed

---

## After completing each unit

1. Run the verification checklist for that unit. Do not mark a unit complete if any checklist item fails.
2. Update `context/progress-tracker.md`:
   - Move the unit from **In Progress** to **Completed**
   - Move the next unit into **In Progress**
   - Add any architecture decisions made during implementation
   - Add session notes if the next session will be picked up by a different person
3. Commit with a message in this format: `feat(U##): short description`

---

## Verification checklist — run before marking any unit complete

- [ ] The feature works end-to-end for the happy path
- [ ] Error states are handled — no blank screens or unhandled promise rejections
- [ ] No TypeScript errors (`tsc --noEmit` passes)
- [ ] No `any` types introduced
- [ ] No raw phone numbers in any log, response body, or database row
- [ ] No hardcoded secrets or API keys in source files
- [ ] The responsible AI label ("AI suggestion") is present on any Gemini output shown in the UI
- [ ] The simulated voice transcript carries its "Simulated for demo" label
- [ ] `progress-tracker.md` is updated

---

## Split work between backend and frontend

When given a unit that touches both backend and frontend:
- Always complete the backend route and verify it works with curl before wiring the frontend.
- State which side you are working on at the start of the implementation.
- Commit the backend work before switching to the frontend.

---

## Gemini prompt rules

Every Gemini prompt must:
1. End with the explicit instruction: `Return only valid JSON. No markdown, no backticks, no explanation.`
2. Include the expected JSON shape in the prompt so the model knows the exact fields.
3. Have a hardcoded fallback in the catch block — never let a Gemini failure crash the webhook handler.

---

## Africa's Talking webhook rules

The webhook handler must:
1. Return HTTP 200 within 5 seconds — AT will retry if it doesn't get a response.
2. Send the 200 response before awaiting the Gemini call.
3. Run the Gemini analysis and Supabase insert asynchronously after the 200 is sent.

```typescript
// Correct order
res.status(200).send()
// Then do the async work
processIncomingSms(from, text).catch(console.error)
```

---

## Forbidden patterns

- Never store raw phone numbers — always hash first with `hash.service.ts`
- Never put Supabase queries in route or controller files
- Never put Gemini calls outside `gemini.service.ts`
- Never use `res.send(error)` or expose raw error objects to the client
- Never use `console.log` — use `console.error` for errors only
- Never add authentication, login flows, or session handling — it is out of scope
- Never add WebSocket / real-time subscriptions — use polling
- Never add features that are listed under "Explicitly out of scope" in `project-overview.md`
- Never label a simulated feature as real in the UI
